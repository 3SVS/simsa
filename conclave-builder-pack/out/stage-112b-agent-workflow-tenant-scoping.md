# Stage 112B — Agent Workflow Record Tenant Scoping

**Date:** 2026-06-23
**Train:** Agent Workflow Train (Stage 110~116) · branch `feat/stage-110-agent-run-plan` · PR #144 (do not merge until Stage 116). Hardening stage before Stage 113.

## Goal
Make the Stage 112 persisted agent workflow records **tenant-scoped and
production-safe** before any future production migration/deploy. Resolves the
Stage 112 production-hardening note (table had no ownership scope).

After this stage:
- saved workflow records are owned by a workspace `user_key`
- list returns only the current scope's records
- detail returns a record only when owned by the current scope
- POST creates records under the current scope
- tests cover cross-tenant isolation

## Tenant scoping decision (follows existing convention)
This repo's workspace API has **no server-side session or header auth**. The
established convention (`workspace_agent_experiments`, `workspace_agent_benchmarks`,
`workspace_credits`, …) is a **client-supplied `userKey`** — an anonymous,
persisted workspace key generated by the dashboard's `getUserKey()`
(`workflow-store.ts`, `localStorage`), sent in the **POST body** and as a **GET
query param**. The server scopes every read/write by that key.

Per the spec ("follow existing conventions rather than inventing a new auth
model; if the convention uses a different scope model, follow it and explain
why"), Stage 112B uses this same model rather than introducing a header/session
identity that does not exist anywhere else in the codebase. The security property
that matters is fully enforced: **a record created under one `userKey` can never
be listed or fetched by another** — the `userKey` is the tenant identifier the
client presents, the record is created under it, and it is only ever read back
under the same key. There is no separate "trusted" identity to override, so
there is no body-vs-trusted mismatch to ignore.

## Route behavior
| Method | Path | Scope behavior |
| --- | --- | --- |
| POST | `/workspace/agent-workflows` | `userKey` required (body) → record inserted under it; `user_key` not echoed in response |
| GET | `/workspace/agent-workflows` | `userKey` required (query) → lists only that user's records; `projectId` filters within that scope |
| GET | `/workspace/agent-workflows/:id` | `userKey` required (query) → returns record only if `record.user_key === userKey`, else **404** (not 403); `user_key` not exposed |

DB module (`workspace/agent-workflow-record-db.ts`): `insertWorkflowRecord` takes
`userKey`; `listWorkflowRecords` requires `{ userKey }` and filters by it;
`getWorkflowRecordById` → `getOwnedWorkflowRecordById` returns the row including
`userKey` so the route can verify ownership and strip it from the response.
