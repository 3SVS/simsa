# Stage 112 — Persisted Agent Workflow Records

**Date:** 2026-06-23
**Train:** Agent Workflow Train (Stage 110~116) · branch `feat/stage-110-agent-run-plan` · PR #144 (do not merge until Stage 116).

## Goal
Persist the deterministic intake workflow snapshot generated by the dashboard
preview flow so it can be **listed and reopened later**. The preview chain is:

```
Intake draft → Type preview → Acceptance Map → Stage Plan → Agent Run Plan → Evidence Plan
```

Stage 112 adds an optional **"Save workflow plan"** step that stores the
acceptance map + stage plan + agent run plan + evidence plan snapshots. Saving is
optional — the preview still works without it.

## Approved scope (narrow, explicit)
Approved by Bae:
- D1 migration for saved intake agent workflows
- central-plane routes to save / list / read saved workflow records
- dashboard save / load UI for Agent Run Plan + Evidence Plan

Not approved (NOT done): actual agent execution · benchmark handoff ·
decision/outcome persistence · evolution action pack creation · GitHub API
execution · Anthropic/Codex/Claude invocation · file upload · screenshot/test
execution · billing/payment · deploy or migration application to production.

## Product principle
This is still **not agent execution**. The feature is a **"Saved workflow plan"**,
never a "Completed agent run", "Executed task", or "Verified result". Statuses:
`draft / planned / needs_evidence / archived`. Avoid `passed / verified /
production_ready`.

## Migration — `0046_workspace_agent_workflow_records.sql`
```sql
CREATE TABLE IF NOT EXISTS workspace_agent_workflow_records (
  id TEXT PRIMARY KEY,
  project_id TEXT,                 -- nullable: intake may begin before a project exists
  intake_type TEXT NOT NULL,
  title TEXT NOT NULL,
  source_summary TEXT NOT NULL,
  raw_input_excerpt TEXT,          -- excerpt only (length-limited), never full blob
  acceptance_map_json TEXT NOT NULL,
  stage_plan_json TEXT NOT NULL,
  agent_run_plan_json TEXT NOT NULL,
  evidence_plan_json TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'planned',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
CREATE INDEX idx_workspace_agent_workflow_records_project ON ...(project_id, created_at DESC);
CREATE INDEX idx_workspace_agent_workflow_records_created ON ...(created_at DESC);
```
JSON snapshots stored as TEXT for D1 compatibility. **Migration not applied to
production in this stage.**

## Central-plane — `routes/workspace-agent-workflow.ts` + `workspace/agent-workflow-record-db.ts`
Browser-facing, CORS-enabled (`corsMiddleware`). IDs prefixed `wawr_`.

| Method | Path | Behavior |
| --- | --- | --- |
| POST | `/workspace/agent-workflows` | Validate + insert; returns 201 with full record |
| GET | `/workspace/agent-workflows` | List summaries (no JSON snapshots); `?projectId=` filter |
| GET | `/workspace/agent-workflows/:id` | Full record with parsed JSON snapshots; 404 if missing |

Validation: required `intakeType` (allowlist), `title`, `sourceSummary`,
`acceptanceMap`, `stagePlan`, `agentRunPlan`, `evidencePlan` (each → 400 with a
specific error). `status` optional → defaults `planned`; unknown value → 400
(no silent coerce). `rawInputExcerpt` trimmed to 2000 chars. Each snapshot must
be an object and ≤ 200 KB (oversized → 400) so huge pasted blobs are not stored.

## Dashboard — `/projects/new/intake`
- **API client** `lib/workspace-agent-workflow-api.ts` — `saveWorkflowRecord`,
  `listWorkflowRecords`, `getWorkflowRecord`; uses

## Stage 112B hardening
- **User/tenant scoping added.** `user_key TEXT NOT NULL` column; every record is
  owned by the workspace key that created it (same model as
  `workspace_agent_experiments` / `workspace_agent_benchmarks`).
- **Migration 0046 updated before production application** — edited in place (no
  0047) to add `user_key` + user-scoped indexes (`(user_key, created_at DESC)`,
  `(user_key, project_id, created_at DESC)`).
- **POST scopes server-side.** This repo's workspace model has no session/header
  auth; the client presents its `userKey` (body for POST, query for GET), exactly
  like the rest of the workspace API. The record is created under that key and is
  only ever read back under the same key.
- **List/detail are scoped.** List returns only the caller's records (projectId
  filter applies within the caller's scope). Detail returns the record only when
  `record.user_key === userKey`, otherwise **404** (not 403 — does not reveal
  another tenant's record exists). `user_key` is never exposed in responses.
- **Cross-tenant isolation tests added** — list excludes other users, detail of
  another user's record returns 404, projectId filter does not leak across users,
  end-to-end "invisible by id" check. central-plane workflow tests: 14 → 20.
- **Production migration still not applied** — no deploy, no remote D1 apply.
