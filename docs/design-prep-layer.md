# 설계 — Simsa "준비 계층" (In-app 셋업 + 배포/DB 커넥터 + Simsa 스킬)

작성: 2026-07-06 · 상태: 제안 (Phase 1 착수 대기, 보안 방식 승인 필요)

## 0. 문제 (왜)

비개발자 사용자가 빌더팩으로 앱을 만들 때, **개발 잡무가 사용자의 코드 에디터로 샌다** — 외부 서비스 가입, API 키 복사, `.env` 작성, 배포, 환경변수 등록, git 절차. 빌더팩 프롬프트(#255)로 "손잡고 안내 / 결과물로 마무리 / 숙제 목록 금지"까지 밀었지만 두 가지 한계가 남는다:

1. **계정 소유자만 할 수 있는 일**(키 재발급, 서비스 로그인)은 에이전트가 대신 못 한다.
2. **호스트 도구(클로드코드 등)의 자체 스킬**(finishing-a-development-branch 등)이 프롬프트를 넘어 개발자 절차를 다시 꺼낸다.

**목표:** 가장 어려운 셋업을 사용자의 에디터가 아니라 **Simsa의 통제된 UX 안에서** 끝내고, 팩을 **배포 준비 완료 상태로** 내보내며, 사용자의 에디터엔 **Simsa가 소유한 워크플로우(스킬)**를 얹어 개발 잡무를 ≈0으로 만든다.

## 1. 세 계층 (아키텍처)

| 계층 | 무엇 | 사용자 숙제 감소 | 난이도 |
|---|---|---|---|
| **A. In-Simsa 셋업 + .env 굽기** | 필요한 서비스를 Simsa 화면에서 가이드로 셋업 → 값 수집 → 팩에 `.env` 포함 | 중 (키를 Simsa UI에서, 에디터 아님) | 낮음 |
| **B. 배포/DB 커넥터 (OAuth)** | Supabase·Vercel OAuth로 Simsa가 DB 프로비저닝·배포를 대신 | 상 (키 복사조차 없음) | 높음 |
| **C. Simsa 스킬 (+ MCP)** | 에디터용 스킬이 build→자가점검→배포→복귀 루프를 소유 | 상 (에디터 잡무 제거) | 중 |

세 계층은 **독립적으로 가치**를 주며(하나만 해도 개선), 합치면 "spec → 준비 완료 팩 → 스킬로 짓기 → 커넥터로 배포 → Simsa 재검수"가 사용자 숙제 없이 돈다.

## 2. 계층 A — In-Simsa 셋업 + .env 굽기 (Phase 1, 근시일·저위험)

### 흐름
1. **서비스 감지** — spec의 포함 기능에서 필요한 서비스를 추론(데이터 저장 → DB, 로그인 → auth, 파일 → storage, 결제/이메일 등). LLM 추론 또는 규칙 기반 + 아이디어 질문 1개("데이터를 저장하나요? 계정 로그인이 필요한가요?", #256 질문 차원과 연결).
2. **가이드 셋업 UI** — 감지된 서비스마다, 초보자 안내(가입 URL → 키 찾는 정확한 위치 → 붙여넣기)를 **Simsa 웹 단계**로. 값은 브라우저에서 입력받는다.
3. **팩에 굽기** — export 시 `.env.local`(실제 값) + `.env.example`(placeholder) + `SETUP.md`(무엇을 왜 넣었는지)를 팩에 포함. → 사용자의 에이전트는 셋업 없이 바로 빌드.

### ★ 보안 (Rule 3 — make-or-break)
- **키를 Simsa 서버에 저장하지 않는다.** 브라우저에서만 보관 → **export 시점에 클라이언트 사이드로 팩의 `.env.local`에 직접 써 넣고** 서버엔 안 남긴다. 저장 책임 = 0.
- `.env.local`(gitignore) / `.env.example`(placeholder) 분리 + "**절대 커밋·공유 금지 · 관리자키는 서버에서만**" 경고를 크게. (팩을 zip으로 공유하면 키가 새므로.)
- `service_role` 같은 관리자 키는 애초에 수집을 최소화하고, 꼭 필요할 때만 + 프론트 금지 경고와 함께.

### 재설계된 프로젝트 흐름 (Bae 지정, 2026-07-06)

준비 계층은 별도 기능이 아니라 **프로젝트 흐름의 "2 검수·준비" 단계**에 녹아든다. dogfooding으로 도출된 목표 흐름:

```
1 준비            아이디어 → 제품설명서(spec) → 확인 항목(items)
   │
2 검수·준비        · 저장소 연결(개발자 경로, 선택)
                  · 알림 설정
                  · 서비스 가입 + API 키 확보·입력   ← 계층 A (준비 계층)
                  · 빌더팩 다운로드                  ← ★ 위로 올림 (여기서 받는다)
   │
[에디터에서 제작]   사용자의 에이전트가 팩으로 빌드
   │
결과 가져오기       배포 URL 또는 프로젝트 파일을 다시 넣기 (/p/{id}/connect 복귀)
   │
3 결과·수정        · 사전확인 / 검수 결과
                  · (수정지시서 ✗ — 여기 다시 둘 필요 없음, 중복 제거)
```

핵심 변경점:
- **"2 검수" 탭에서 바로 빌더팩으로 가면 안 된다.** 먼저 연결·알림·서비스/키 입력을 하고, **빌더팩 다운로드도 이 단계로 올린다**(현재 하단/별도 export였음).
- 전체 루프를 **"다 연결 → 팩 다운 → 에디터 제작 → 결과(URL/파일) 다시 넣기 → 결과·수정"** 으로 다시 세팅.
- **수정지시서(fix brief)는 "3 결과·수정"에서 제거** — 중복(이미 팩/다른 곳에 있음).
- (버그) 코드확인에서 코드변경 연결 후 "확인하기"를 누르면 **"2 코드변경"으로 되돌아가고 기존 연결이 풀린다** → 별도 수정 필요(아래 §9).

이 흐름 재배치는 `project-steps.mjs` 상태머신 + 사이드바 nav + export UI 위치 이동을 건드리는 별도 작업이다(준비 계층 구현과 함께 P1로).

### 구현 지점
- `apps/central-plane/src/workspace/export.ts` — `.env.example` + `.env.local` 템플릿 + `SETUP.md` 생성 (값 주입은 클라이언트에서).
- 대시보드 — 서비스 감지 결과 + 가이드 셋업 스텝(신규 화면/모달). export API가 브라우저 수집 값을 받아 팩에 합침.
- #255 프롬프트의 "에이전트가 Supabase 셋업 손잡기"는 **Simsa가 미리 채운 서비스엔 불필요**해지고 폴백으로만 남음 → 자연스러운 시너지.

## 3. 계층 B — 배포/DB 커넥터 (Phase 2/3, 고가치·고난이도)

### Supabase 커넥터 (DB 프로비저닝)
- Simsa가 사용자 Supabase org에 **OAuth 연결** → 프로젝트 생성 + **팩에 담긴 마이그레이션 SQL 적용** + anon 키 조회를 Simsa가 대신. **사용자는 `service_role` 키를 복사할 일이 없다.**
- 저장은 **암호화된 OAuth 토큰만**(원시 DB 키 아님).

### Vercel 커넥터 (배포)
- 사용자 Vercel에 OAuth 연결 → 프로젝트 import/배포 + 환경변수 설정 + 배포 URL 반환. "배포" 버튼 한 번.

### 결과
"내 데이터베이스 준비"와 "배포"가 Simsa 안에서 원클릭 → 사용자 숙제 거의 0. = Bae의 "여기까지 다 준비돼있어야지"의 실체.

### 보안·의존성
- OAuth 토큰은 **기존 GitHub 토큰 암호화 패턴(`CONCLAVE_TOKEN_KEK`, AES-GCM) 재사용**. 스코프 최소, 사용자 취소 가능.
- 파트너 의존: Supabase/Vercel **OAuth 앱 승인·약관**. 이게 리드타임.
- 스코프 크리프 경계: 모든 서비스 지원 금지. **DB(Supabase) + 배포(Vercel)만** 먼저.

## 4. 계층 C — Simsa 스킬 (+ MCP)

### 무엇
클로드코드/커서용 **공식 Simsa 스킬(`SKILL.md`)** 이 루프를 소유:
1. spec·수용기준·(계층 A/B의) env를 **기존 MCP(Stage 61 `packages/mcp-workspace`)로 끌어온다**.
2. 에이전트에게: **수용기준에 맞춰 빌드**(셋업 잡무 없음, env 이미 준비) → **자가 점검** → **배포**(커넥터 또는 가이드) → **Simsa로 복귀**(#243 딥링크). = D1-b 루프를 **스킬이 강제**.
3. **비개발자 마무리 규칙**(git 절차 질문 금지, 결과물로 끝, 숙제 목록 금지)을 **스킬 레벨에서** 강제 → 팩 프롬프트보다 강함(호스트의 generic 스킬을 대체하므로).

### MCP 확장
`packages/mcp-workspace`에 도구 추가: 빌더팩 조회 · env 조회 · 검수 제출(submit-for-review) · 배포 트리거(커넥터 경유).

### 배포(distribution)
Simsa에 **"내 에디터에서 쓰기 / 클로드코드 연결"** 문서 한 장(스킬 1개 + MCP 설정). 스킬 라이브러리가 아니라 **Simsa 자체를 스킬로**.

### 전제
- 스킬을 깔아 쓰는 사람은 이미 완전 초보가 아님(Bae 관찰) → 이 계층의 우선순위는 "완전 초보용 준비물"보다 낮음. 다만 **팩 프롬프트 한계(호스트 스킬 override)를 근본적으로 푸는** 유일한 방법.

## 5. 로드맵 (순서)

1. **P1 — 계층 A** (근시일, 저위험, 가치 80%): 서비스 감지 → Simsa 내 가이드 셋업 → **무저장 .env 굽기**. ※ 보안 방식(무저장+브라우저 주입) 승인 필요.
2. **P2a — 계층 C 스킬** (`SKILL.md` + MCP 확장 + "에디터에서 쓰기" 페이지): 팩을 진짜 스킬로, 비개발자 마무리를 스킬로 강제.
3. **P2b — 계층 B Vercel 커넥터** (배포 원클릭). 파트너 OAuth 필요.
4. **P3 — 계층 B Supabase 커넥터** (DB 프로비저닝). 최고 난이도, 파트너 OAuth.

## 6. 이미 있는 것 (위에 얹는다)

- **MCP 서버** (Stage 61, `packages/mcp-workspace`) — 계층 C 기반.
- **D1-b 회귀 훅 + `/p/[id]/connect`** (#243) — 복귀 루프.
- **빌더팩 export** (`export.ts`) — 여기에 env/마이그레이션/배포 타깃 추가.
- **GitHub 토큰 암호화** (`CONCLAVE_TOKEN_KEK`, AES-GCM) — 계층 B 토큰 저장 패턴.
- **초보 셋업 가이드** (#255) — 계층 A가 못 채운 서비스의 폴백.
- **UIUX/서비스 질문** (#256) — 계층 A 서비스 감지의 입력.

## 7. 리스크 · 결정 사항

- **보안 (최우선, Rule 3):** P1은 원시 키 무저장(브라우저 주입). P2는 암호화 OAuth 토큰만(원시 키 아님). → **결정 필요: P1 무저장 방식 승인?**
- **파트너 의존:** Supabase/Vercel OAuth 앱 승인·약관 (P2 리드타임).
- **스코프:** DB(Supabase)+배포(Vercel)만 먼저. 모든 서비스 지원 금지.
- **유지보수:** 가이드 클릭 경로가 낡음 → 한 데이터 파일에 모아 쉽게 갱신.

## 9. 알려진 버그 (별도·근시일 수정)

- **코드확인 연결 상태 유실:** 코드확인 화면에서 코드변경(PR)을 연결하고 "확인하기"를 누르면 **"2 코드변경" 단계로 되돌아가고, 기존에 연결해 둔 것도 풀린다**. 연결 상태가 유지되지 않거나(재조회 stale) 진행 맵이 hasRepo=false로 오판해 되돌리는 것으로 추정. 재현 후 원인(연결 저장/재조회 vs project-steps 판정)을 특정해 수정 + 회귀 테스트.

## 8. 한 줄 요약

프롬프트로는 "떠넘기지 마라"까지가 한계다. **셋업을 Simsa 안으로 가져오고(A), 배포·DB를 커넥터로 대신하고(B), 에디터엔 Simsa 스킬을 얹어(C)** 사용자 개발 잡무를 0에 수렴시킨다. P1(무저장 .env 굽기)부터, 보안 방식 승인 후 착수.
