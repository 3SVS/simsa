# 온보딩 · 실현가능성 레이어 (설계)

> 상태: 설계 확정, 단계별 구현 착수. Bae 지시(2026-07-07~08).
> 배경: PISTA(Kotlin Android + Firebase)에 빌더팩을 돌렸더니 "Next.js 웹앱 →
> Vercel 배포"를 가정한 껍데기가 나왔다. Simsa가 **유저를 프로파일링하지 않고**,
> **플랫폼을 확정하지 않고**, **할 수 있는 것/없는 것을 솔직히 말하지 않은** 결과.

## 원칙 — 정확하고 솔직하게

Simsa는 "다 된다"고 말하지 않는다. 아이디어/PRD를 읽고 **플랫폼별로 무엇을
정확히 해줄 수 있는지 솔직하게** 판정한다.

- 웹앱: "이건 웹앱으로 구현 가능합니다. **A+B 조합**을 추천합니다." → 끝까지(빌드·배포·실코드 검수)
- 모바일: "모바일 앱을 희망하시는군요. 심사에서 **정확한 구현까지는 어렵고**,
  개발자·에이전트에 **전달하실 수 있는 자료**(스펙·항목·점검 리포트)까지
  만들어드립니다." → 핸드오프 산출물, 가짜 웹앱 배포 금지

## 1. 온보딩 인터뷰 (프로젝트 시작 시 · 프로파일 먼저)

비개발자 대상인데 유저가 어떤 사람인지 묻지 않고 개발자 가정으로 진행하던 것을
고친다. 시작 화면에서 캡처:

| # | 질문 | 선택지 | 쓰임 |
|---|---|---|---|
| 1 | 무엇을 만들고 싶으세요? | 자유 입력 (아이디어/PRD) | 실현가능성 판정 입력 |
| 2 | 어디서 돌아가는 앱인가요? | 웹사이트·웹앱 / 휴대폰 앱 / 잘 모르겠어요 | 플랫폼 seed (LLM 판정과 교차) |
| 3 | GitHub 써보셨어요? | 있고 익숙 / 들어봤지만 잘 몰라요 / 처음 들어요 | 안내 강도·용어 |
| 4 | AI 코딩 도구(Cursor 등) 써보셨어요? | 네 / 조금 / 아뇨 | 안내 강도 |
| 5 | (초보 선택 시) 계정 만들기부터 도와드릴까요? | 네 / 이미 있어요 | 계정 준비 플로우 |

레벨은 우선 **3단계(초보 / 중간 / 능숙)**. 필요하면 세분화한다(Bae 승인).

## 2. 실현가능성 판정 (LLM이 아이디어/PRD를 읽고 분류)

`idea-to-spec` 생성 앞단(또는 병행)에 **feasibility 분류 단계**를 둔다. 입력은
아이디어/PRD + 인터뷰 답(플랫폼). 출력은 카테고리 + 솔직한 메시지 + 추천.

| 카테고리 | 판정 | Simsa가 하는 것 | 솔직 메시지(예) |
|---|---|---|---|
| `web_buildable` | 웹앱으로 구현 가능 | 스펙·항목·**빌더팩**(플랫폼 맞춤)·**스택 추천(A+B)**·배포·실코드 검수 | "이건 웹앱으로 구현 가능합니다. **{A}+{B}** 조합을 추천합니다." |
| `mobile_handoff` | iOS/Android 네이티브 | 스펙·항목·**전달용 자료**(PRD·점검 리포트·수용 기준). 웹 배포·Vercel 안내 **금지** | "모바일 앱을 희망하시는군요. 정확한 구현까지는 어렵고, 전달하실 수 있는 자료까지 만들어드립니다." |
| `other_handoff` | 데스크톱·게임엔진·하드웨어·불확실 | 할 수 있는 범위 판정 + 핸드오프 자료 + **범위 밖 항목 명시** | "이 부분은 심사 범위 밖이라, 대신 {산출물}을 만들어 전달을 돕겠습니다." |

- **추천 스택(A+B)**: web_buildable일 때만. 지금처럼 Next/Vercel/Supabase를
  무조건 박지 않고, 아이디어에 맞는 조합을 **추천**하되 사용자가 바꿀 수 있게.
- 판정이 애매하면 **물어본다**(인터뷰 Q2). 확신 없이 밀어붙이지 않는다.

## 3. 코딩 레벨이 안내를 조절 (플랫폼과 직교)

| 레벨 | 적용 |
|---|---|
| 초보 | "GitHub이 뭔지"부터 설명, 계정 없으면 **만들기부터**, 전문용어 0, 가드레일 최대 |
| 중간 | 핵심만 설명, 선택적 도움말 |
| 능숙 | 설명 스킵, 빠른 경로 |

## 4. 데이터 모델

```
// 프로젝트에 부착 (extended data 또는 project)
userProfile: {
  platform: "web" | "mobile" | "unknown";
  githubLevel: "fluent" | "heard" | "new";
  aiToolLevel: "yes" | "some" | "no";
  needsAccountHelp: boolean;
  codingLevel: "beginner" | "intermediate" | "advanced"; // 3·4에서 파생
}
feasibility: {
  category: "web_buildable" | "mobile_handoff" | "other_handoff";
  message: string;            // 솔직 메시지 (locale)
  recommendedStack?: string[]; // web_buildable일 때 A+B
  outOfScope?: string[];       // 핸드오프일 때 범위 밖 항목
}
```

하위 소비처는 이 둘을 읽어 분기한다:
- `mcp-catalog` / `service-catalog`: platform=web일 때만 Vercel/`NEXT_PUBLIC_*`
  기본. 그 외엔 강요 금지(현 `detectMcpTools()` 무조건 추가 제거).
- 빌더팩 생성(central-plane `export.ts`): 카테고리별 산출물·배포 안내 분기.
- 검수/연결 플로우: 핸드오프 카테고리는 "저장소 연결→실코드 검수" 대신
  전달 자료 다운로드로 유도.

## 5. 구현 단계

- **Phase 1 — 인터뷰 + 프로파일 캡처**: 시작 화면 질문(위 5개), `userProfile`
  저장. 아직 하위 분기는 없음(기본=web, 현행 유지). i18n EN/KO.
- **Phase 2 — 실현가능성 판정**: LLM feasibility 분류(게이트웨이 경유·정직
  실패), 솔직 메시지·추천스택 산출, 결과 화면. 카테고리 저장.
- **Phase 3 — 하위 분기**: service/mcp 카탈로그·빌더팩·플로우가 platform·
  category·codingLevel을 읽어 분기. Next/Vercel/Supabase 무조건 제거.
- **Phase 4 — 핸드오프 산출물**: mobile/other 카테고리 전용 전달 자료
  (PRD·점검 리포트·수용 기준) 생성 + 다운로드.

각 Phase는 base=main 독립 PR, 원격 CI green 후 머지, 수동 배포.

## 6. 안 하는 것 (범위 명확화)

- Simsa가 모바일 네이티브 코드를 직접 생성/빌드/배포하지 않는다(솔직).
- 추천 스택은 **강요가 아니라 제안** — 사용자가 바꿀 수 있다.
- 확신 없는 자동 판정으로 밀어붙이지 않는다 — 애매하면 묻는다.
