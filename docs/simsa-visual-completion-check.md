# Simsa Visual Completion Check + 비개발자 한국어 리포트 (Stage 260A)

> **누구를 위한 것인가:** 바이브코딩으로 앱을 만든 **한국인 비개발자**. "내 앱이 진짜 작동하나?"를
> 개발자 없이, **눈으로 직접 보면서** 확인하고, 안 되면 **무엇을·왜·어떻게 고치는지** 평범한 한국어로
> 받는다.
>
> **상태:** 코드-레디니스. 순수 로직은 `apps/central-plane`(테스트+CI 포함), 실제 브라우저 구동은
> `tools/simsa-completion-loop-spike/visual-run.mjs`. 프로덕션 배포·golf-now 수정 없음.

## Stage 258A 대비 무엇이 바뀌었나

258A 스파이크는 첫 화면에서 **버튼 하나**만 눌러봤다. 비개발자가 원하는 "이거 작동해?"에는 부족했다.
260A는 세 가지를 메운다:

1. **깊은 실사용 플로우** — 버튼 하나가 아니라, 안전한 핵심 CTA를 누르거나 **검색창에 실제로 입력**하고
   결과 화면까지 관찰한다 (`visual-flow-plan.ts`, 다단계).
2. **눈으로 보는 증거** — 단계마다 스크린샷 + **진행 영상(webm)** 을 남긴다.
3. **비개발자용 한국어 리포트** — `ERR_NAME_NOT_RESOLVED` 같은 용어를 **무엇이/왜/어떻게** 3줄로
   번역하고, 원본 기술 문자열은 "개발자용" 칸에만 둔다 (`nondev-report.ts`).

## 왜 URL만 찍어보는 QA로는 부족한가

서버가 `200 OK`를 줘도 사용자가 아무것도 못 할 수 있다. golf-now 실제 배포본이 그렇다 — 홈은 200이지만
코스 데이터 백엔드가 죽어 있어 **골퍼가 코스를 검색·확인할 수 없다.** "작동하냐?"는 HTTP 상태가 아니라
**실제 사용자 흐름**의 속성이다. 그래서 진짜 브라우저로 **의도(intent anchor) 기준** 흐름을 밟는다.

## 왜 의도 기준(intent anchor)이 필요한가

의도가 없으면 "다 됐냐?"는 답할 수 없다 — *무엇을 위해* 됐냐? 의도 앵커("골퍼가 지금 코스가 플레이
가능한지 확인하는 흐름을 시작할 수 있어야 한다")가 잣대다. 플래너는 이 의도에 맞는 동작을 고른다.

**의도 정렬 예시:** golf-now 홈에는 "🛡️ 비 보험 가입하기" 버튼도 있다. 단순 키워드 매칭이면 `가입`을
보고 그 버튼을 누른다 — 하지만 의도는 "코스 확인"이므로 잘못된 선택이다. `intentIsSearchOriented` +
`ctaIsSearchLike` 규칙이 이 경우 **가입 버튼 대신 '골프장 검색' 입력창을 쓰도록** 강제한다.

## Browser Evidence vs AI Opinion (섞지 않는다)

- **Browser Evidence** = 사실만: 로드 상태, 클릭/입력한 것, 이동한 주소, 콘솔 오류, 실패한 네트워크
  요청, 스크린샷, 타임스탬프.
- **비개발자 리포트** = 사실을 사람 말로 번역한 것. 각 항목의 원본 오류코드는 "개발자용" 칸에만. 판정은
  **점수가 아니라 상태**(작동해요 / 작동 안 해요 / 직접 확인 필요 …).

## 판정(점수 없음)

`Ready 정상 작동해요` · `Needs Fix 작동 안 해요` · `Not Verified 확인 못 했어요` · `User Acceptance
Required 직접 눈으로 확인이 필요해요` · `Needs Clarification 무엇을 확인할지 애매해요` …

**숫자 점수는 없다** (Simsa 정책 §20). 깨끗하게 흐름이 돌아도 스파이크에는 "이 화면이 진짜 쓸 만한가"를
판정할 시각 오라클이 없으므로 **자동으로 Ready라고 하지 않고** "직접 눈으로 확인 필요"에서 멈춘다.

## 산출물

`report.html` — 자체완결 한국어 페이지. 비개발자가 더블클릭하면: 판정 배지(✅/⛔/⚠️) → 한 줄 요약 →
무엇이/왜/어떻게 카드 → 화면 스크린샷 → 진행 영상 → 다음에 할 것. 개발자용 기술 정보는 접힘(details).
함께 `report.md`, `report.json`, `browser-evidence.json`도 남는다.

## 데모 (golf-now, 읽기 전용 대상)

`conclave-builder-pack/out/stage-260a-visual-completion-korean-report/`:

- `demo-golf-now-live/` — **실제 배포본(고장)**. 검색창에 "서울" 입력 → 결과 안 뜸. 판정 **작동 안 해요**;
  이유를 한국어로("데이터 서버 주소를 못 찾음") + 개발자용에 원본 `ERR_NAME_NOT_RESOLVED`.
- `demo-golf-now-repaired-local/` — Stage 258C 오프라인 폴백이 적용된 **로컬 수정본**. 같은 검색이
  결과를 띄우고 **에러 0**. 판정 **직접 눈으로 확인이 필요해요** (검색은 됐고, 최종 사용성만 사람이 확인).

이 둘을 나란히 보면 비개발자도 "고치기 전 vs 후"를 눈으로 안다.

## 한계 (명시)

- 하나의 핵심 흐름만 밟는다 (홈 → 핵심 CTA/검색 → 다음 화면).
- **시각 오라클 없음:** 이동/에러/입력 성공 같은 사실은 확인하지만, 다음 화면이 "정말 쓸 만한지"는
  사람이 최종 확인해야 한다.
- 로그인/결제/삭제/발행/배포 같은 위험·비가역 동작은 하지 않는다. auth 우회 없음.
- 모든 버그를 찾는다는 뜻도, 제품이 완벽하다는 뜻도 아니다.

## 실행

```
pnpm --filter @simsa/central-plane build     # 순수 모듈 빌드 (tool 이 dist 를 import)
cd tools/simsa-completion-loop-spike && npm install # 로컬 격리, 브라우저는 캐시 사용
node visual-run.mjs <targetUrl> <outDir> [검색어]
```
